## Executive Summary

On February 26, 2026, a sustained SSH brute-force attack was detected against endpoint **ubuntu12** (10.128.124.88), originating from **10.128.124.9** (Kali Linux — simulated threat actor). The attack resulted in **1,391+ failed authentication attempts** targeting the root account over 15 minutes.

The attack was detected within **2 minutes** by Splunk correlation rule SOC-001, automatically triaged by Shuffle SOAR, enriched via VirusTotal API, and a Jira incident ticket (SOC-2) was auto-created within **45 seconds** of detection.

## Lessons Learned

### What Worked Well ✅

| Item | Impact |
|------|--------|
| Detection time under 2 minutes | Fast response enabled |
| SOAR reduced manual response by 90% | Efficiency gain |
| Jira ticket auto-created in 45 seconds | Zero documentation lag |
| MITRE mapping identified full attack chain | Complete visibility |
| Threat hunting found additional IOCs | Proactive coverage |

### What Needs Improvement ⚠️

| Issue | Recommendation |
|-------|---------------|
| SSH on standard port 22 | Move to non-standard port |
| No fail2ban installed | Install on all endpoints |
| VirusTotal limited to public IPs | Add public IP simulation |
| Alert fatigue from 1391 events | Implement rate limiting |

### Recommendations

1. **Implement fail2ban** on all Linux endpoints — auto-blocks after 5 failures
2. **Disable root SSH login** permanently via `PermitRootLogin no`
3. **Add GeoIP blocking** for non-India source IPs
4. **Increase Splunk index retention** to 90 days for forensics
5. **Schedule weekly threat hunting** reviews using TH-001 through TH-005
